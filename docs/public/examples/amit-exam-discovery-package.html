Problem Statement

Before any design or architecture work began, the team explored who would use this feature and what problem it actually solves. What follows is not a summary of the client's requirements — it is a record of what was observed, what was concluded, and what the client confirmed is their genuine pain.

Problem Statement

Validated

When a teacher needs to run a formal summative assessment for their class, they are forced to use external tools because the Amit LMS has no exam capability. Assessment results exist outside the system where all other learning activity is tracked. The teacher manually transfers grades between systems, cannot see assessment data alongside learning data, and loses the connection between what students learned and how they performed.

Client confirmed: "Yes — this is our actual pain. Teachers spend approximately 3 hours per exam on manual grade transfer alone. The feature is not about a new assessment tool. It is about connecting assessment to learning and eliminating manual grade transfer entirely."

Experience Snapshot — Gal, Creating

G
Gal — Science Teacher
Middle school · 2 years using Amit LMS
Validated
Who
Gal. Middle school science teacher. Two years using the Amit LMS. Comfortable with the course builder for lessons. Has distributed the science curriculum to her class of 28 students. Has never created a formal exam in this system. Her mental model for exams comes from Google Forms or printed Word documents.
Situation
Two weeks before end of term. Creating a summative exam covering three curriculum units. Wants a 45-minute total time limit. Three sections for three units. Wants to prevent students returning to earlier sections. Wants automatic submission when time runs out. Building the exam the afternoon before it is given. She has two hours. She is alone in the teachers' room.
Goal
Create a 45-minute three-section exam with 15 questions, publish it to her class for tomorrow morning, and trust that submissions and grades will flow to the school system without manual re-entry.
Current Experience
Google Forms with a third-party timer add-on. Shares the link via class WhatsApp. Manually checks who submitted. Downloads a response spreadsheet. Enters scores into a separate grade spreadsheet. Enters the final grade into the Education Management System (EMS) separately. Total time: approximately 3 hours per exam including marking.
What Success Looks Like
Gal creates the exam in under 30 minutes. Sets the timer without confusion. Publishes to her class. The next morning all 28 submissions arrive with auto-checked scores for multiple choice. She marks open-ended responses in one view, publishes grades, and they reach the EMS automatically. Total time: under 36 minutes.
What Failure Looks Like
Gal is confused by three simultaneous timer options and sets all three without understanding their interaction. One student loses internet mid-exam and she cannot find the retry option. Students submit with unanswered questions and she cannot tell if this was intentional. Grades do not reach the EMS automatically and she enters them by hand as before.

Experience Snapshot — Gal, Grading

G
Gal — Grading Mode
Same teacher · The exam ran yesterday
Hypothesis
Who
Gal. Same teacher. The exam ran yesterday. 28 students submitted.
Situation
Wednesday morning free period. 45 minutes available. 28 submissions to review. Five open-ended questions per submission — 140 manual marks needed. One student (Maya) lost internet mid-exam and submitted with three empty answers.
Goal
Review all 28 submissions, handle Maya's exception, publish grades to students and the EMS — all before end of day.
Current Experience
No exam-specific review exists. Gal opens individual submissions, marks manually, calculates totals per student by hand, enters grades into the EMS separately. No aggregate class view. Cannot see which questions the class struggled with. Cannot compare answers to the same question across students.
What Success Looks Like
Results view shows all 28 students with auto-calculated scores for multiple choice already done. She works through open-ended responses. Sees a class performance summary showing which questions were most commonly missed. Releases a retry for Maya in two clicks. Publishes all grades at once. Confirmation that grades were synced to the EMS.
What Failure Looks Like
Individual exercise scores with no exam aggregate. Manual total calculation per student. Maya's submission is hard to find. Retry release buried in an admin panel she has never used. Grade publication fails silently and she discovers the EMS did not receive grades three days later.
Critical Finding
Gal spends 30 minutes creating an exam. She spends 3 hours grading it. The original specification spent three pages on creation and one sentence on grading. The grading journey is the higher-value journey. Without it the feature does not move KR1.

ADR-001 — Timer Entity Model

ADR-001 — Timer Entity Model

Architecture Decision Record
Context
The specification describes timers at exam, section, and page level simultaneously. No data model exists for timer configuration. No interaction model defines what governs when multiple timers are active. Eight of twelve possible timer interaction states were undefined.
Decision
Timer configuration is stored as a nullable JSON column (timer_config) on Topic, Section, and Page entities. A boolean field (is_exam) on Topic gates all timer behavior. Lesson topics are never affected — timer fields are read only when is_exam = true.
Timer Interaction Rule
When multiple timer levels are active, the most restrictive timer governs at any moment. Page timer overrides section timer overrides exam timer. This resolves all eight previously undefined states.
Migration
Three additive ALTER TABLE statements. No existing data is affected. Rollback: drop the three timer_config columns and the is_exam column.
Consequences
Builder UI must check is_exam before rendering timer controls. Player must read timer config from all three entity levels on session start. A timer summary view in the builder settings panel is required.

ADR-002 — Exam State Persistence Model

ADR-002 — Exam State Persistence Model

Architecture Decision Record
Context
The specification requires local draft saving on every answer change AND disconnection recovery. Together these imply continuous server-side state persistence. The current player uses local-only exercise state which cannot survive device restart or network loss.
Decision
Exam state is persisted server-side on every answer change (debounced at 2 seconds). New table: exam_session, storing student_id, exam_id, current_page, answer_state (JSON), timer_state (JSON), started_at, last_saved_at, submitted_at, submission_status.
Disconnection Recovery
On reconnect, the player fetches the last exam_session record. Timer is recalculated from started_at minus total elapsed at last save. Student lands on their last active page with all answers intact.
Timer Security
Timer is authoritative server-side. Client display only. Discrepancies over 30 seconds flag the submission for teacher review.
Performance
28 simultaneous students at average 30-second answer intervals produce approximately 14 server writes per minute. Within current infrastructure capacity.
Consequences
New exam_session table required. Composite index on (student_id, exam_id). The continuous write endpoint carries an explicit performance SLO of P99 < 200ms.

ADR-003 — Exam Grading Architecture

ADR-003 — Exam Grading Architecture

Architecture Decision Record
Context
The specification delegates exam grading to the existing exercise checking system in one sentence. Investigation found: the checking system handles individual exercise submissions only — it does not support bulk exam submission. The assessment profile does not accommodate exam-specific scoring. EMS integration expects a specific grade schema that has never received exam data.
Decision
The existing checking system is extended with a new exam_submission endpoint accepting bulk submission. New table: exam_result, aggregating per-student results: total_score, max_score, auto_checked_score, manual_score, status, published_at, ems_sync_status.
Assessment Profile
Exam topics bypass the standard assessment profile calculation. Exam scores are calculated directly from exam_result.
Manual Grading
Open-ended responses flagged requires_manual_review = true. Teacher grading view filters for these. Marking updates exercise_response and triggers exam_result recalculation.
EMS Integration
Exam results are sent to the EMS as grade type assessment_exam. Schema confirmation from EMS API team is pending. This is the one external dependency blocking Slice 3.
Review App
A source_type column (lesson / exam) is added to exercise_response. The review app adds a source filter. Impact: 1 day of development.
Consequences
Two new tables. Extended checking system endpoint. EMS integration is a hard dependency for Slice 3 only — Slices 1 and 2 deliver full value without it.

is_exam on Topic

ALTER TABLE topic ADD COLUMN is_exam BOOLEAN DEFAULT false; -- Gates all exam-specific UI and behavior. -- Rollback: ALTER TABLE DROP COLUMN

exam_session (new table)

CREATE TABLE exam_session ( id UUID PRIMARY KEY, student_id UUID NOT NULL, exam_id UUID NOT NULL, started_at TIMESTAMP NOT NULL, last_saved_at TIMESTAMP, submitted_at TIMESTAMP, submission_status ENUM('active', 'submitted', 'expired', 'recovered'), current_page_id UUID, answer_state JSON, timer_state JSON ); CREATE INDEX idx_session_student_exam ON exam_session (student_id, exam_id); CREATE INDEX idx_session_submitted ON exam_session (submitted_at); -- Rollback: DROP TABLE exam_session

exam_result (new table)

CREATE TABLE exam_result ( id UUID PRIMARY KEY, exam_id UUID NOT NULL, student_id UUID NOT NULL, total_score DECIMAL, max_score DECIMAL, auto_checked_score DECIMAL, manual_score DECIMAL, status ENUM('pending', 'pending_manual_review', 'complete'), published_at TIMESTAMP, ems_sync_status ENUM('not_sent', 'pending', 'synced', 'failed'), ems_sync_at TIMESTAMP ); CREATE INDEX idx_result_exam_student ON exam_result (exam_id, student_id); -- Rollback: DROP TABLE exam_result

source_type on exercise_response

ALTER TABLE exercise_response ADD COLUMN source_type ENUM('lesson', 'exam') DEFAULT 'lesson'; UPDATE exercise_response SET source_type = 'lesson'; -- Enables review app filter. -- Rollback: ALTER TABLE DROP COLUMN

API Contract — All Endpoints

POST /exams

Create exam topic. Auth: course_editor or teacher role. Returns exam_id and default config.

PUT /exams/:id/config

Update timer configuration and navigation rules. Auth: course_editor or teacher (own course only). Validates timer interaction rules server-side.

GET /exams/:id/session

Get or create student exam session. Auth: student, must be enrolled. Returns existing session or creates new exam_session with status = active.

PUT /exams/:id/session

Continuous state write. Debounced client-side at 2s. Auth: student. SLO: P99 < 200ms. Updates answer_state and timer_state on exam_session. Performance-critical endpoint.

POST /exams/:id/submit

Final submission. Auth: student. Server validates timer has not expired independently. If race condition with timer expiry: server processes first request, returns 409 Conflict to duplicate. Creates exam_result.

GET /exams/:id/results

All student results for this exam. Auth: teacher who owns the course. Returns student list with status, scores, final status.

GET /exams/:id/results/:studentId

One student's full response detail. Auth: teacher. Returns all exercise_response records with marking status.

PUT /exams/:id/results/:studentId/mark

Add manual score to one open-ended response. Auth: teacher. Validates score within max_score. Triggers exam_result recalculation.

POST /exams/:id/results/publish

Publish all grades. Auth: teacher. Sets published_at. Triggers EMS sync if Slice 3. Returns sync status.

POST /exams/:id/students/:studentId/retry

Release retry attempt. Auth: teacher. Preserves previous session as Attempt 1. Creates new exam_session. Teacher sets custom timer duration.

Key Design Decisions Made During Shaping

1. Submit does not block on unanswered questions
The spec said "must answer all questions to submit" but also "timer expiry submits empty answers." Contradiction. Resolution: manual submit is always available. Popup warns about unanswered. Student decides. Auto-submit on timer expiry always submits.
2. Timer summary view required in builder
Teachers set all three timer levels without understanding interaction. Summary panel shows all active timers and governing rule.
3. Retry timer is teacher-configurable
Spec said timer resets to full. Client confirmed: teachers should set custom duration. Full reset creates unfair advantage.
4. Question-centric marking view deferred to Slice 3
Found in grading Snapshot. Teachers want to mark all answers to question 7 together. Client confirmed. Added to Slice 3.
5. Concurrent submit and timer expiry resolved
Race condition: timer expires at exact moment student clicks Submit. Server processes first request, returns 409 Conflict to duplicate. Prevents double submission.
